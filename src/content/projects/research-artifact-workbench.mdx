## Question

What should a project page prove before a reader opens the repository?

This example treats the project page itself as an artifact. The goal is to make
the shape of future research work obvious: claim first, evidence second,
implementation third, limitations always visible.

## Artifact contract

| Section | Purpose | Reader question |
| --- | --- | --- |
| Claim | State what the artifact is for | What is being asserted? |
| Workflow | Show the reproducible path | How would I rerun it? |
| Evidence | Summarize output and checks | What did it produce? |
| Limits | Bound the interpretation | Where can this fail? |

## Minimal workflow

The command block is intentionally concrete. Even if the command is only a
template today, future projects should aim for this level of operational clarity.

```bash
pnpm run collect --target tls-boundary-slice
pnpm run normalize --input results/raw
pnpm run report --format static
```

```ts
export type EvidenceRow = {
  target: string;
  observation: string;
  confidence: "low" | "medium" | "high";
  artifactUrl: string;
};

export function isReviewable(row: EvidenceRow) {
  return row.confidence !== "low" && row.artifactUrl.length > 0;
}
```

## Limitations

- A project page is not a substitute for a repository.
- A visual summary should not hide uncertainty.
- A strong artifact page should make missing work visible, not embarrassing.

## Why keep this page

This is a useful prototype for future project pages. If a real artifact cannot
fill this skeleton, the artifact probably needs more work before it becomes the
centerpiece of an application or lab conversation.
