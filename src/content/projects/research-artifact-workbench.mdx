## Question

What evidence does the page expose before the reader opens the repository?

This example treats the project page itself as an artifact. Future research pages
use this shape: claim first, evidence second, implementation third, scope visible
throughout.

## Artifact contract

| Section | Purpose | Reader question |
| --- | --- | --- |
| Claim | State what the artifact is for | What is being asserted? |
| Workflow | Show the reproducible path | How would I rerun it? |
| Evidence | Summarize output and checks | What did it produce? |
| Limits | Bound the interpretation | Where can this fail? |

## Minimal workflow

The command block is concrete by design. The template gives future projects an
operational path for collection, normalization, and reporting.

```bash
pnpm run collect --target tls-boundary-slice
pnpm run normalize --input results/raw
pnpm run report --format static
```

```ts
export type EvidenceRow = {
  target: string;
  observation: string;
  confidence: "low" | "medium" | "high";
  artifactUrl: string;
};

export function isReviewable(row: EvidenceRow) {
  return row.confidence !== "low" && row.artifactUrl.length > 0;
}
```

## Scope notes

- Repository links remain the source for implementation detail.
- Visual summaries carry uncertainty through scope notes.
- Missing work appears as open questions and assumptions.

## Template use

This prototype sets the shape for future project pages. A real artifact fills the
skeleton with commands, outputs, links, and scope notes before it becomes a main
research object.
