This portfolio uses three surfaces with different jobs: notes, projects, and
papers or manuscripts. The separation keeps early observations, working artifacts,
and stronger claims in the right place.

The distinction is simple.

| Surface | Job | Evidence |
| --- | --- | --- |
| Notes | Record observations and technical choices | sketches, measurements, implementation details |
| Projects | Present the artifact | code, traces, datasets, reproducible workflows |
| Papers | State the supported claim | model, method, related work, evaluation, limits |

Notes can be smaller and sharper. A note can record a boundary worth measuring,
a fragile protocol behavior, or an invariant ready for formalization.

Project pages carry stricter structure:

- What question was being tested?
- What artifact exists?
- What would convince a skeptical reader?
- What scope is claimed?

Papers sit at the end of that path. They carry the strongest claim the artifact,
method, and evaluation can support.

## Artifact record

A useful artifact page exposes the object under study, the method, and the
evidence available for inspection.

```yaml
artifact:
  question: "What behavior is being measured?"
  evidence:
    - traces
    - normalized dataset
    - reproducible command
  failure_modes:
    - sampling bias
    - parser assumptions
    - deployment drift
```

This structure separates claim, evidence, and scope. A broad claim with thin
evidence becomes visible; a strong artifact can stay quiet and still read clearly.

```ts
type ResearchOutput = {
  claim: string;
  artifact: "note" | "project" | "paper";
  evidence: Array<"code" | "trace" | "model" | "measurement" | "review">;
  limitations: string[];
};
```

> A page earns space when it answers the next technical question.

## Reading path

For lab and PhD readers, the path is topic, note, artifact, stronger claim. The
reader sees the problem class, the method, the implementation object, and the
evidence supporting the current scope.

For engineering readers, the same structure exposes the operable object: code,
commands, outputs, links, and maintenance surface.
